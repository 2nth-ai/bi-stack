Overview

What this is and why it exists.

bi-stack is a self-hosted Business Intelligence deployment built on open-source primitives: Apache Superset for the dashboarding layer, dbt for transformation, and Neon serverless Postgres as both the Superset metadata store and the analytics warehouse. Celery + Redis provide async work; Cloudflare Pages iframes the UI into client demos using Superset guest tokens.

The repo ships with two fully modelled data domains so the same stack can be demoed against very different shapes of data:

  • SAFEX commodity hedging — the original 2nth.ai domain: clients, contracts, futures prices, basis/margin math, hedge coverage. Schema: analytics_analytics (see the note on the dbt schema-concat quirk below).
  • Retail DC (Shoprite-style) — a large FMCG retailer’s distribution-centre inventory: 8 DCs, 2,493 SKUs, 820 stores, 183k pallet positions, 553k movements. Schema: analytics_retail_dc.

The retail-DC domain exists to demonstrate that the stack handles a broad product range and high-cardinality inventory data, not just narrow financial tables.

Architecture

Components and the boundaries between them.

Why these boundaries

  • Region split. Cloud Run runs in africa-south1 (Johannesburg) for SA-first latency. Neon sits in eu-central-1 (Frankfurt) because that is its closest region; cross-region latency is ~150 ms, acceptable for analytical queries. Upstash Redis is in af-south-1 (Cape Town) for ~5 ms cache round-trips from Cloud Run.
  • Two Postgres endpoints on the same DB. Superset’s metadata DB uses the direct (unpooled) Neon endpoint because transaction-level pooling breaks prepared statements and Alembic migrations. dbt uses the pooled endpoint for its many short-lived connections.
  • Separate metadata and analytics schemas. superset_meta holds dashboards, charts, users, roles. analytics + analytics_retail_dc are the warehouse data. Superset registers the warehouse schemas as "database connections" of its own metadata instance, which is a legitimate Superset pattern even when both schemas live on one physical DB.
  • dbt runs externally. dbt doesn’t run inside Superset — it is invoked either locally or from a Cloud Run Job on a Cloud Scheduler cron. Its only job is to produce the analytics_* tables; Superset only reads.

Schema-concat quirk. dbt-postgres’s default generate_schema_name concatenates the profile’s target schema with any model-level +schema: override, so schema: analytics plus +schema: analytics in dbt_project.yml produces analytics_analytics rather than plain analytics. Stack choices

What each piece is doing and why.

Local setup

Clone-to-running in about ten minutes on a fresh macOS.

# 1. Clone and check out the merge branches
gh repo clone 2nth-ai/bi-stack
cd bi-stack
gh pr checkout 1          # first-run bring-up fixes (not yet merged)
gh pr checkout 2          # retail DC demo (not yet merged)

# 2. Generate .env from a Neon pooled URI.
#    secrets.txt lives only on disk (gitignored, mode 600).
# Paste your Neon pooled URI into secrets.txt (format shown in .env.example)
# e.g. scheme://USER:PASS@ep-<id>-pooler.<region>.aws.neon.tech/neondb?sslmode=require
./scripts/setup-env.sh

# 3. One-time schema init on Neon (use the SQL Editor in the Neon console)
cat sql/neon-init.sql
# paste the CREATE SCHEMA statements into Neon and run

# 4. Bring up the stack
docker compose up --build -d
# wait for bi-superset-init to exit 0, then:
open http://localhost:8088
# admin / (see ADMIN_PASSWORD in .env)
Admin password. Generated by setup-env.sh (base64-encoded 24 bytes). Read it back with grep ADMIN_PASSWORD .env. The same credentials work on the Cloud Run deploy because the metadata DB is shared.

Run dbt against Neon

python3 -m venv .venv
.venv/bin/pip install dbt-core==1.8.7 dbt-postgres==1.8.2

# extract NEON_USER and NEON_PASSWORD from the pooled URI, then:
export DBT_PROFILES_DIR=./dbt
.venv/bin/dbt deps
.venv/bin/dbt seed    # SAFEX seeds
.venv/bin/dbt run
.venv/bin/dbt test

Retail DC demo

Adding the second domain with ~740k rows of synthetic retailer data.

The generator uses nothing but Python’s standard library and a fixed random seed, so every run produces identical data. Dimension tables are kept in Git; fact tables are regenerated on demand to keep repo size sane.

# 1. Generate all CSVs
python3 scripts/generate_retail_dc.py
#   dbt/seeds/dc/dim_*.csv        (committed)
#   dbt/seeds/dc/fct_pallet_stock.csv  (183k rows, gitignored)
#   scripts/out/fct_stock_movements.csv (553k rows, gitignored)

# 2. Seed the small dims + pallet stock via dbt (takes ~3 minutes for pallet_stock)
.venv/bin/dbt seed --select path:seeds/dc

# 3. Bulk-load the 553k-row movements fact via psycopg2 COPY (~8 seconds)
.venv/bin/python scripts/load_movements.py

# 4. Build the DC models
.venv/bin/dbt run  --select path:models/dc
.venv/bin/dbt test --select path:models/dc

# 5. Register Superset dataset + charts + starter dashboard
.venv/bin/python -u scripts/superset_bootstrap.py

Pareto velocity

Outbound movements are weighted so the top 20 % of SKUs per DC carry ~80 % of volume. This produces a believable inventory health distribution (~22 % stockout risk, ~21 % healthy, ~31 % overstock) rather than a flat "everything has infinite cover" demo.

GCP deploy

One-shot Cloud Run deploy in africa-south1.

export PATH="/opt/homebrew/share/google-cloud-sdk/bin:$PATH"
gcloud auth login
gcloud auth application-default login

# Create a dedicated project, link billing
gcloud projects create nth-bi-stack --name="bi-stack" --set-as-default
gcloud billing projects link nth-bi-stack --billing-account=XXXXXX-XXXXXX-XXXXXX
gcloud config set project nth-bi-stack

# 1. APIs, Artifact Registry, service accounts, empty secrets, WIF for CI
export GCP_PROJECT=nth-bi-stack
bash gcp/secrets-bootstrap.sh

# 2. Populate Secret Manager from local .env (plus an Upstash rediss:// URL)
for k in SUPERSET_METADATA_DB_URI ANALYTICS_DB_URI NEON_POOLED_URI \
         SUPERSET_SECRET_KEY GUEST_TOKEN_JWT_SECRET ADMIN_PASSWORD; do
  grep "^$k=" .env | cut -d= -f2- | tr -d '\n' \
    | gcloud secrets versions add "$k" --data-file=-
done

# 3. Build + push Superset image (linux/amd64 for Cloud Run)
gcloud auth configure-docker africa-south1-docker.pkg.dev --quiet
docker buildx build --platform linux/amd64 \
  -t africa-south1-docker.pkg.dev/nth-bi-stack/bi-stack/superset:latest \
  --push ./superset

# 4. Deploy Cloud Run service (web tier only — see gotcha below)
bash gcp/deploy-web.sh

# 5. Enable public access (needs an org-policy override for 2nth.ai)
gcloud services enable orgpolicy.googleapis.com
# write a project-level override that allows allUsers bindings, then:
gcloud run services add-iam-policy-binding bi-superset-web \
  --region=africa-south1 --member=allUsers --role=roles/run.invoker
Worker / beat not deployed. Cloud Run Services require an HTTP listener on $PORT. Celery worker and beat don’t serve HTTP, so gcp/deploy-worker.sh and gcp/deploy-beat.sh fail with "container failed to start and listen on the port". For dashboard viewing this is irrelevant — Celery only matters for async SQL Lab, scheduled reports and thumbnails. Proper fix: a supervisord + tiny HTTP stub wrapper, or move worker/beat to a Compute Engine micro-VM.

Result

Public URL: https://bi-superset-web-<project-number>.africa-south1.run.app/login/ (HTTP 200). Dashboard is at /superset/dashboard/retail-dc-ops/. Admin login is the same as the local .env value because the Superset metadata DB is shared.

First-run gotchas

Four runtime issues caught by the initial bring-up that are now fixed in the image.

SymptomRoot causeFix
ModuleNotFoundError: No module named 'flask_cors' during superset init ENABLE_CORS = True in superset_config.py but flask-cors was not in requirements-local.txt Pinned flask-cors==4.0.1 in superset/requirements-local.txt
Every API call 422: Subject must be a string Flask-AppBuilder 4.5.0 emits int JWT subjects; PyJWT ≥2.10.1 rejects them on decode (the CVE-2026-32597 fix introduces this strictness) Pinned flask-appbuilder>=4.5.4,<4.6
SSL SYSCALL error: EOF detected ~17 min into the init permission loop Superset init runs a long transaction touching many permission rows; Neon’s proxy drops idle TCP connections mid-transaction without keepalives Added connect_args with keepalives=1, keepalives_idle=30, keepalives_interval=10, keepalives_count=5 to SQLALCHEMY_ENGINE_OPTIONS
_gdbm.error: [Errno 13] Permission denied: 'celerybeat-schedule' Celery beat writes its schedule file to CWD, which is /app in the image — read-only for the superset user Added --schedule=/tmp/celerybeat-schedule to the beat args in docker-compose.yml and gcp/deploy-beat.sh

Operations

Day-two tasks.

Refresh the data

# Re-seed pallet stock + reload movements + rebuild marts
python3 scripts/generate_retail_dc.py
.venv/bin/dbt seed --select fct_pallet_stock --full-refresh
.venv/bin/python scripts/load_movements.py
.venv/bin/dbt run  --select path:models/dc
.venv/bin/dbt test --select path:models/dc

Redeploy a new image to Cloud Run

docker buildx build --platform linux/amd64 \
  -t africa-south1-docker.pkg.dev/nth-bi-stack/bi-stack/superset:latest \
  --push ./superset
bash gcp/deploy-web.sh
# Cloud Run will create a new revision and shift 100% traffic once healthy.

Tear down

gcloud projects delete nth-bi-stack   # destroys everything
# Neon and Upstash are managed separately: delete those databases from their consoles.
